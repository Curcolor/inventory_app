¿Qué es una Dependencia?

Definición
Dependencia

Cualquier objeto o clase que otra clase necesita para funcionar. Sin ella, la clase no puede operar.

Sin DI ✗

La clase crea su propia dependencia internamente. Acoplamiento fuerte.

Con DI ✓

La dependencia llega desde afuera. La clase solo la usa, no la fabrica.

❌ Sin DI — acoplamiento fuerte dart
1class AuthService {
2 // La clase CREA su propia dependencia
3 final HttpClient client = HttpClient();
4}
✅ Con DI — inyección por constructor dart
1class AuthService {
2 final HttpClient client;
3
4 // La dependencia LLEGA desde afuera
5 AuthService(this.client);
6}
7
8// Uso:
9final service = AuthService(HttpClient());

¿Qué es la Inyección de Dependencias?

Definición

Patrón donde los objetos reciben sus dependencias desde el exterior en lugar de crearlas.

3 Problemas que Resuelve

① Acoplamiento fuerte

Sin DI, cambiar una dependencia obliga a modificar la clase que la usa.

② Testing imposible

Sin DI no puedes pasar un fake o mock para pruebas unitarias.

③ Instancias duplicadas

Cada clase crea la suya propia: múltiples conexiones abiertas sin darte cuenta.

Provider en Flutter

Base
InheritedWidget

Mecanismo nativo de Flutter para compartir datos en el árbol sin pasar parámetros manualmente.

Provider es un wrapper sobre
InheritedWidget

Simplifica la creación, consumo y disposición automática de dependencias.

Principios

  • Todo es un widget — el estado vive en widgets padre
  • Flujo top-down — datos de padres a hijos
  • BuildContext — mecanismo de acceso en O(1)

Árbol de Widgets y Flujo de Datos

Raíz
Global / MultiProvider

Registra InventoryRepository e InventoryProvider. Accesibles para todo el árbol.

Acceso al BuildContext global. Cualquier widget descendiente puede localizar los providers.

Pantalla
Scaffold / Page

Puede tener su propio ChangeNotifierProvider local, independiente del estado global.

Rebuild selectivo: solo se reconstruye el widget que necesita actualizarse.

Sin estado propio. Solo recibe datos o callbacks. UI limpia y testeable.

paquete provider

Componentes Clave del Paquete Provider

Provider<T>

Expone un valor inmutable al árbol. Crea y destruye el valor con create / dispose. Úsalo para servicios que no notifican cambios.

ChangeNotifierProvider<T>

Escucha un ChangeNotifier y reconstruye widgets dependientes cuando se llama notifyListeners(). Es el más usado para estado reactivo.

Consumer<T>

Obtiene el valor del provider con un BuildContext descendiente. Se reconstruye solo cuando el ChangeNotifier llama notifyListeners().

Selector<T, R>

Como Consumer pero más granular: reconstruye solo cuando cambia la propiedad seleccionada. Evita rebuilds innecesarios.

MultiProvider

Compone una lista de providers sin anidación profunda. Hace el código plano y legible en lugar de un árbol de widgets anidados.

ProxyProvider<A, B>

Construye un provider que depende de otro ya registrado. Se actualiza automáticamente. Ej: ProxyProvider<Repo, Service> inyecta el Repo en el Service.

08 / 12
casos reales

Problemas que Resuelve

01 — Prop Drilling
Pasar datos por capas intermediarias

Problema: datos que viajan por 5 widgets aunque solo el último los necesita.

Solución: Provider ubica el servicio en el widget padre adecuado; cualquier descendiente lo consume directamente con context.watch.

02 — Estado Global Acoplado
Variables globales o singletons

Problema: variables globales dificultan las pruebas y el mantenimiento. Imposible reemplazar en tests.

Solución: Provider maneja instancias de forma organizada y permite reemplazarlas fácilmente en pruebas.

03 — Rebuilds Innecesarios
Todo el árbol se reconstruye

Problema: cambiar un único campo reconstruye toda la pantalla, lo que es costoso en renders.

Solución: Selector<T,R> escucha solo la propiedad seleccionada, limitando el rebuild al mínimo necesario.

04 — Ciclo de Vida
Fugas de memoria

Problema: liberar streams y controllers manualmente es propenso a errores y fugas de memoria.

Solución: el dispose() del Provider se ejecuta automáticamente al salir el widget del árbol.

Intermediario de datos

Solo busca y devuelve datos. No usa ChangeNotifier porque no maneja estado reactivo — solo opera bajo demanda.

¿Por qué JSON + SharedPreferences?

SharedPreferences solo guarda tipos simples (String, int, bool). Un Product es complejo, por eso se empaqueta como JSON String para poder guardarlo.

Lectura

conclusión

Provider-Based DI en Flutter

01
Provider es la forma nativa de Flutter para DI, construida sobre InheritedWidget. Sin librerías externas pesadas.
02
Resuelve prop drilling, estado global acoplado y rebuilds innecesarios de forma elegante.
03
La separación Repository → Provider → UI da bajo acoplamiento y alta testabilidad.
04
Selector limita los rebuilds al mínimo. MultiProvider evita el nesting profundo.
05
Entender Provider facilita adoptar Bloc o Riverpod, que se construyen sobre el mismo patrón.
